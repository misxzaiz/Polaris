# Product

## Register

product

## Users

Polaris serves developers and AI-assisted coding users who work across multiple projects, terminals, repositories, and AI coding engines. They use the product in focused engineering sessions on desktop and mobile companion surfaces, often switching between chat, code context, tasks, Git state, plugins, and scheduler workflows.

## Product Purpose

Polaris provides a unified graphical workspace for multiple AI coding engines and developer automation tools. Its app-side experience should make existing desktop capabilities approachable on mobile: resume conversations, monitor agent work, inspect project context, operate MCP/plugin tools, manage todos/scheduler items, and stay oriented across workspaces without relying on command-line interaction. Success means users can confidently continue, review, and steer engineering work from an iOS-style mobile interface with minimal friction.

## Brand Personality

Precise, calm, capable. The interface should feel like a high-trust developer cockpit adapted for mobile: refined, fast, legible, and quietly powerful rather than decorative or consumer-chat-like.

## Anti-references

Avoid generic AI chat clones, over-decorated glassmorphism, low-density toy dashboards, marketing-style hero metric layouts, loud gradients, and desktop UI compressed directly into a phone viewport. Avoid interfaces that hide operational state, make engine/workspace context ambiguous, or prioritize visual novelty over task clarity.

## Design Principles

1. Preserve technical confidence: expose the active engine, workspace, session state, and tool progress clearly.
2. Make continuation effortless: prioritize recent sessions, resumable tasks, pending decisions, and next actions.
3. Use mobile-native hierarchy: replace desktop panes with focused cards, bottom navigation, sheets, and progressive disclosure.
4. Keep actions close to context: chat controls, file references, Git changes, and scheduler actions should appear where the user needs them.
5. Design for long-running work: agent runs, background sessions, scheduler logs, and plugin services need compact but trustworthy status language.

## Accessibility & Inclusion

Target WCAG AA contrast for text and controls. Support reduced motion, large touch targets, system font scaling, clear focus states, non-color-only status indicators, and legible dark/light appearances. Motion should communicate state changes only and should not delay task completion.
