# 需求文档模板

> AI 生成需求时必须遵循此模板结构。所有章节按顺序排列，必填章节不可省略。

---

## 必填章节

### 1. 问题概述（1-2 句话）

描述现状和痛点，聚焦「是什么问题」而非「怎么解决」。

示例：当前文件编辑器关闭已修改文件时未弹出确认对话框，可能导致用户丢失未保存的工作内容。

### 2. 背景分析

引用相关代码和现有实现，说明当前状态和为何需要改进。必须引用至少一个相关文件路径。

格式要求：
- 使用行内代码标注文件路径：`src/path/to/file.ts:行号`
- 如无行号可省略，但路径不可省略
- 说明当前实现状态和局限性

示例：

- `src/stores/fileEditorStore.ts:119` — Tab 关闭逻辑，直接移除无确认
- `src/components/Layout/CenterStage.tsx` — TabBar 组件，未检查 dirty 状态

### 3. 改进目标

使用数字编号列表（1. 2. 3.），每项有明确描述。子项用无序列表（-），最多 2 层嵌套。

示例：

1. **追踪文件修改状态**
   - 扩展 Tab 接口添加 isDirty 字段
   - 内容变化时自动标记为已修改
2. **关闭确认对话框**
   - 提供保存/不保存/取消三个选项
   - 支持批量关闭时的处理逻辑
3. **视觉反馈**
   - 已修改 Tab 显示 dirty 指示器

### 4. 影响范围

列出涉及的文件路径，每行一个，使用行内代码格式 + 简要说明。

示例：

- `src/stores/fileEditorStore.ts` — 扩展 Tab 接口和 dirty 状态管理
- `src/components/Layout/CenterStage.tsx` — 关闭 Tab 前检查 dirty 状态
- `src/components/Common/UnsavedDialog.tsx` — 新建确认对话框组件

### 5. 预期效果

3-5 条简洁要点，描述改进后的效果。

示例：

- 防止用户意外丢失编辑内容
- 提升数据安全性和用户信心
- 与主流编辑器行为一致

---

## 选填章节

以下章节根据具体情况选用：

### 技术方案（如需特定技术选型说明）

描述实现所需的技术选型、外部依赖、架构变化等。

### 原型说明（如生成了原型文件）

说明原型文件路径和关键设计决策。

---

## JSON 安全规范

description 字段写入 requirements.json 时必须遵循：

1. **禁止 ASCII 双引号**（U+0022）：使用中文引号「」或 `\"` 转义
2. **禁止原始换行符**：使用 `\n` 表示换行
3. **文件路径使用正斜杠** `/`：不使用反斜杠 `\`
4. **JSON 缩进 2 空格**
5. **长文本注意转义**：反斜杠 `\\`、制表符 `\t` 等特殊字符需正确转义

---
